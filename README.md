# 巡天御风 2025 视觉组第三次考核任务书

## 前言

GitHub 是我们学习、分享技术的重要平台，也是团队协作与项目管理的核心工具。本次考核分为两个维度：一是通过填空题巩固现代 C++ 核心语法；二是通过「GitHub 保卫战」这一特殊玩法，切实练习 Git/GitHub 的版本管理与协作流程。希望大家在完成考核的过程中，既能加深对 C++ 的理解，也能熟练掌握 GitHub 的基本操作。

---

## 任务内容

### 目标 1 · 现代 C++ 填空题（30 分）

仓库 `exam/questions/` 目录下共有 **10 道填空题**，涵盖以下 C++11/14/17/20 特性，请逐一补全每道题中标注 `TODO` 的空白处，使所有测试断言通过：

| 题号 | 主题                        | 关键知识点                                                    |
| ---- | --------------------------- | ------------------------------------------------------------- |
| Q01  | 自动类型推导                | `auto`、`decltype`、尾置返回类型                          |
| Q02  | Lambda 表达式               | 捕获列表、`mutable`、泛型 lambda                            |
| Q03  | 智能指针                    | `unique_ptr`、`shared_ptr`、`weak_ptr`                  |
| Q04  | 移动语义与完美转发          | 右值引用、移动构造、`std::forward`                          |
| Q05  | 结构化绑定                  | `auto [a, b]`、对 map/pair/struct 解包                      |
| Q06  | `optional` 与 `variant` | `nullopt`、`value_or`、`std::visit`                     |
| Q07  | `constexpr` 与编译期求值  | `constexpr` 函数/变量、`consteval`、`if constexpr`      |
| Q08  | Concepts 与约束             | `concept`、`requires`、缩写函数模板                       |
| Q09  | 现代类特性                  | `override`/`final`、`= default`/`= delete`、委托构造  |
| Q10  | Ranges 与视图               | `std::ranges::sort`、`filter`/`transform`/`take` 管道 |

> 编译环境：CMake ≥ 3.20，C++20，GCC/Clang 均可。
> 构建命令：
>
> ```bash
> cd exam
> cmake -B build && cmake --build build
> ctest --test-dir build --output-on-failure
> ```

### 目标 2 · GitHub 代码保卫战（50 分）

这是本次考核的特殊玩法，流程如下：

#### 阶段一：自主解题并提交 PR（截止 **3 月 10 日 24:00**）

1. **Fork** 本仓库到你自己的 GitHub 账号。
2. 在 Fork 后的仓库中新建文件夹（可参考之前的要求，文件夹命名只需要包含***姓名***即可），完成所有填空题。
3. 确保代码可以正常编译，后续会创建自动化action实时验证编译状态。
4. 向本仓库发起 **Pull Request**，PR 标题格式：`[解题] 姓名 - 第三次作业 - 初始代码`。

> ⚠️ **3 月 10 日 24:00 之前未提交 PR 目标 2 得 0 分。**

#### 阶段二：发现并修复破坏（存档截止 **3 月 14 日 24:00**）

1. 在 **3 月 10 日**，会对已合并 PR 中的代码进行**一次**可被 Git 追踪的随机破坏（均为***可被git追踪***的破坏，不会出现直接覆盖远程git仓库以直接删除提交记录及类似的极端情况）。
2. 你需要尽快**发现**被破坏的内容，并在本仓库的 **Issues** 中详细描述：
   - 哪个文件 / 哪个位置被破坏；
   - 破坏前后的差异（可附 `git diff` 截图或代码块）；
   - 你判断被破坏的依据。
3. 随后发起新的 **Pull Request** 修复该破坏，PR 标题格式：`[修复] 你的姓名 - 破坏修复`。
4. 在 **3 月 14 日 24:00** 之前，确保你 Fork 仓库中的代码已恢复到破坏前的正确执行逻辑。

> 若最终代码无法恢复到破坏前的状态，将按照恢复程度酌情扣分。
> 修复方法越巧妙（如善用 `git log`、`git bisect`、`git diff` 等），得分越高。
> 虽然可以直接撤销我的commit，但我会放几个雷，如果那部分东西没了查到是谁直接无脑撤销会扣分，

撰写一份 **GitHub 使用学习报告**，要求：

- 内容包含但不限于：Git 常用命令、分支管理、PR 流程、遇到的问题与解决方法；

## 评分细则

| 目标                    | 分值            | 说明                               |
| ----------------------- | --------------- | ---------------------------------- |
| 目标 1 · C++ 填空题    | 30 分           | 按通过测试用例数量比例给分         |
| 目标 2 · GitHub 保卫战 | 50 分           | 见下方详细评分点                   |
| 目标 3 · 学习报告      | 20 分           | 内容完整性与表达质量               |
| **及格线**        | **60 分** | 总分 ≥ 60 且目标 2 ≥ 30 方可及格 |

**目标 2 详细评分点：**

1. PR 提交规范性与时效性（按时提交 PR）。
2. 编译产物完整性（二进制文件是否一并提交）。
3. Issue 的发现与描述质量（定位准确、描述清晰）。
4. 修复 PR 的正确性（代码是否完整恢复）。
5. 修复方法的巧妙程度（是否善用 Git 工具溯源、定位）。

> **早交奖励**：3 月 10 日之前每提前一天提交 PR，额外加 **2 分**（计入日常积分）。
> 请仔细检查代码正确性后再提交，避免无效 PR，出现空 PR 酌情扣分。
